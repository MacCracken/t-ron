# Security Audit — 2026-04-14

**Scope:** t-ron 2.0.0 (Cyrius port), including vendored libro 1.0.3
and bote 1.9.2 as they are built into the released artifact.

**Goal:** CVE / 0-day-class bypass review with repair. Focus on
published attack classes from 2024-2025 MCP / WAF / audit-log
ecosystem research.

**Authors:** Claude (research + implementation), reviewed against live
t-ron sources.

---

## Methodology

1. **External research** — WebSearch + WebFetch over public advisories,
   CVE entries, and vendor disclosures relevant to t-ron's attack
   surface (MCP, injection scanners, JSON/TOML parsing, audit chains,
   rate limiting, UUID generation, supply chain). Output: a 23-item
   prioritized checklist with CVE references. See §Appendix for sources.
2. **Internal review** — line-by-line read of
   `src/{gate,policy,rate,scanner,pattern,audit,tron,middleware}.cyr`
   plus vendored `lib/libro_entry.cyr`, matched against each external
   finding. Grep hints from the checklist drove the search.
3. **Repair** — patches applied for every actionable finding. Each
   patch carries a regression test in `tests/t-ron.tcyr`.
4. **Verification** — full test suite (266 tests / 67 groups) and
   benchmark replay (no >10 % regression on any metric).

## Critical findings — detailed

### F1 — `tools/list` descriptions are not scanned (tool-poisoning)

**Class:** MCP tool poisoning (Invariant Labs, 2025).
**Attack:** A malicious MCP server embeds instructions inside a tool's
`description` field. The LLM reads that description during capability
discovery and obeys the hidden prompt before any user approval step
(Invariant reports 84 % success rate under auto-approval).

**t-ron position:** `security_gate_dispatch` in `src/middleware.cyr`
only intercepts `tools/call`. `tools/list` flows straight through to
bote. This is **by design** — t-ron is a runtime gate, not a registry
validator.

**Disposition:** Documented. The correct layer for mitigation is
either:
1. **Description pinning** — consumers hash each tool's description on
   first approval and reject on subsequent mismatch (tool-shadowing
   defense). Should live in bote's registry, not t-ron.
2. **Static review** — `tron_policy` only references tool names, not
   descriptions; admins vet descriptions out-of-band.

**Follow-up:** Raise with bote team to add `description_sha256` to
`ToolDef` with a `registry_register_pinned(def, expected_sha)` path.

## High findings — fixes

### F2 — libro UUID fail-open on `/dev/urandom` short read

**Class:** CVE-2015-8851-style UUID predictability.
**Location:** `lib/libro_entry.cyr::uuid_v4` (libro 1.0.3 upstream).
**Root cause:** `file_read(fd, buf, 16)` return value ignored. If
`/dev/urandom` fails to open (seccomp, sandbox) or returns short, the
UUID is derived from whatever bytes the freelist allocator returned —
potentially all zeros or a reused page. Combined with the hash chain
linking every entry, a predictable UUID lets an attacker synthesize a
plausible-but-forged chain.

**Patch:** Check `fd >= 0` and `got == 16`; abort with exit code 74
(`EX_IOERR`) and a stderr message on failure. `syscall(60, 74)` is
atomic — no corrupted chain state escapes.

**Regression test:** Implicitly covered by all chain-based tests
passing — any short-read scenario now terminates the process rather
than silently producing degenerate UUIDs.

**Upstream:** Vendored patch in `lib/libro_entry.cyr`; drop when libro
ships a release with the fix.

### F3 — Control-char injection into identifiers (rate-bucket / policy collision)

**Class:** Key-delimiter collision (generic WAF bypass class).
**Location:** `src/rate.cyr::_rate_bucket_key` keys by
`agent_id ++ 0x1F ++ tool_name`. If `agent_id` contains `0x1F` (ASCII
unit separator) or unprintable bytes, two distinct (agent, tool) pairs
can map to the same bucket — one agent's quota drains another's. A
newline in `agent_id` also breaks policy logs.

**Patch:** New gatekeeper `tron_is_safe_identifier(s)` in
`src/gate.cyr` rejects anything with a byte `< 0x20`, `0x7F`, or
`>= 0x80`, plus empty / oversized (> 256 bytes) strings. Called at the
top of `tron_check` for both `agent_id` and `tool_name`. Failure
produces an `Unauthorized` deny, recorded in the audit log — fail-closed
regardless of `default_unknown_agent`.

**Regression test:** `test_audit_unsafe_ids_rejected` — 0x1F, newline,
DEL, non-ASCII, empty all rejected; plain identifiers pass; pipeline
denies end-to-end on 0x1F injection.

**Side benefit:** Also blocks the overlong-UTF-8 bypass in identifiers
(non-ASCII rejected).

### F4a — MySQL versioned comments (libinjection-style bypass)

**Class:** WAF.NINJA / Novikov libinjection bypass.
**Payload:** `/*!50000 UNION */`, `/*!50000SELECT */ … /*!50000FROM*/`.
Every MySQL engine executes these; scanners that tokenize on whitespace
miss them because the keywords are "inside" a comment that MySQL
unwraps.

**Patch:** New `_sc_has_sql_evasion` in `src/scanner.cyr` runs before
keyword matching. If the sequence `/*!` appears anywhere in the
parameter leaf, flag directly as SQL. The pattern has no legitimate
use in a tool parameter.

### F4b — Null-byte keyword splitting

**Class:** WAF tokenizer fooling (`uni\x00on`, `se\x00lect`).
**Payload:** Databases strip nulls; our keyword matcher would have
treated the string as two fragments and missed.

**Patch:** `_sc_has_sql_evasion` also flags any null byte (`0x00`)
inside a parameter leaf. Tool params legitimately never contain embedded
nulls (JSON strings escape them as `\u0000`, which would arrive as
those four characters, not a literal null).

### F4c — Overlong UTF-8 path traversal (`0xC0 0xAE 0xC0 0xAE 0x2F`)

**Class:** SEC Consult Airlock advisory; CVE-2024-34078.
**Payload:** `0xC0 0xAE` decodes to `.` under lenient UTF-8 handlers
(RFC 3629 forbids it, but many string-normalization paths accept it
anyway). Our byte-oriented scanner originally only matched the ASCII
pattern `0x2E 0x2E`.

**Patch:** `_sc_check_path` now walks the buffer for `0xC0 0xAE 0xC0 0xAE`
explicitly. Defense-in-depth: identifier F3 check already blocks non-
ASCII in agent/tool names.

### F4d — Shell metacharacter filler (`${IFS}`, `$9`, brace expansion)

**Class:** HackTricks / PayloadsAllTheThings shell bypass.
**Payloads:**
- `data;${IFS}rm -rf /` — `${IFS}` expands to space, shell executes `rm`
- `data;$9rm x` — `$9` expands to empty (no positional), shell executes `rm`
- `data;{rm,-rf,/}` — brace expansion into `rm -rf /`

Original scanner jumped from the delimiter (`;`, `&`, `|`, `` ` ``,
`$`) over whitespace only; `${IFS}` was opaque, `$9` consumed no
filler, `{…}` opening brace wasn't recognized.

**Patch:** `_sc_skip_shell_filler` now consumes:
- Whitespace
- `${…}` with matching close brace
- `$<digit>` (single digit, positional)
- `$<letter|_>+` (variable name, letters/digits/underscores)
- Opening brace `{` (brace group)

After the filler, the shell-binary match resumes. Positional args
consume *exactly one digit* — critical for catching `$9rm` where the
filler ends at the `9` and the `rm` is exposed.

### F5 — Double-encoded path traversal

**Class:** CVE-2024-38819 class (Spring Framework).
**Payload:** `%252e%252e%2f` — decoders that run twice produce
`..%2f` then `../`. Our literal scan for `%2e%2e` missed the pre-
decoded form.

**Patch:** `_sc_check_path` now also matches `%252e%252e` and the
fully-hex-encoded `%25%32%45%25%32%45` variant. Case-insensitive.

## Medium findings — fixes

### F6 — Template scanner coverage gaps

**Added delimiters:**
- `{% … %}` — Jinja2 block statements (for / if / assign)
- `<# … >` — FreeMarker directives
- `#{ … }` — ERB / Ruby string interpolation

Each extends `_sc_check_template` with the same keyword-trigger
heuristics as the existing `{{ … }}`.

### F7 — Template `{{…}}` close-scan quadratic blowup

**Class:** CVE-2020-28493 (Jinja2 ReDoS).
**Root cause:** old code scanned from each `{{` to EOF looking for
`}}`, falling through and advancing by 2 on failure. N unclosed `{{`
openers → O(N²) total work.

**Patch:** Refactored to `_sc_find_delim2` + `_sc_scan_paired_regions`
helpers. If `}}` isn't found after the current `{{`, bail out of the
entire scan — no further complete regions can exist in this string.
Each byte is now visited O(1) times across the full template scan.

**Regression test:** `test_audit_template_redos` builds a 4096-byte
payload of unclosed `{{` openers, calls `scanner_scan_text`, and
asserts elapsed < 100 ms. On the pre-fix scanner this took seconds; on
the fixed scanner it's microseconds.

## Informational & already-safe

### F8 — JSON duplicate-keys canonicalization
bote parses `params` into cstrings; t-ron scans `params` as raw bytes
without key-aware processing; libro canonicalizes only the `details`
object that t-ron *constructs itself* (not attacker-controlled). No
divergence point between attacker input and audit-chain hashing, so
Bishop Fox's interoperability bypass does not apply.

### F9 — Rate-limiter clock skew
`bucket_refill` already guards against `elapsed_ns <= 0` by resetting
`last_refill` to `now` without adding tokens. Clock-going-backward
scenarios are handled; no patch needed.

### F10 — Ring-buffer eviction of escalation pattern
An attacker who controls call ordering can emit 100 benign calls to
flush the 3-of-5-sensitive window before it triggers. Moving to an
exponential-decay model would close this, at the cost of extra state
and tuning complexity. Flagged for v2 — the current implementation
still catches naive probing, which is the typical threat model for
audit tripwires.

### F12 — Integer overflow in scanner length math
`strlen` is bounded by `SSIZE_MAX` on Linux; intermediate `i + nlen`
sums where both are non-negative and < 2^62 can never wrap in practice.
No exploitable overflow path identified.

### F13 — libro 1.0.3 `_cjh_hash_object` SEGV
Pre-existing vendored patch (see [memory note](../../.claude/memory/libro_1_0_3_bug.md)).
Included here for auditability.

## Out of scope (per the external checklist)

- **XML / XXE (CVE-2024-8176 libexpat):** t-ron parses only JSON + TOML,
  no XML codepath.
- **TLS / mTLS:** below t-ron's layer; bote's concern.
- **SSRF:** t-ron never originates outbound requests.
- **Cross-server tool shadowing (CVE-2025-6514 mcp-remote RCE):**
  description pinning is bote's responsibility, not t-ron's (see F1).
- **Trusting-Trust Cyrius bootstrap:** Diverse Double-Compiling is a
  toolchain concern; track at the Cyrius project, not t-ron.
- **JSON-RPC batch amplification:** bote's dispatch layer, not t-ron.
